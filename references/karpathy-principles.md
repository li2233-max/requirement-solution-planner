# Karpathy Principles For Planning

这份 skill 吸收了 `andrej-karpathy-skills-main` 里最有价值的四个核心原则，但把它们从“编码护栏”翻译成了“需求规划护栏”。

## 核心内容总结

### 1. Think Before Coding -> 先暴露假设，再出方案

原始重点：

- 不要代替用户偷偷做关键假设
- 不要把困惑藏起来
- 有多种解释时要显式展开
- 更简单的路线存在时要说出来

在本 skill 里的映射：

- 先判断是单需求还是多需求
- 先写判断依据和关键假设
- 对存在歧义的输入，先列解释路径和影响，再给默认建议

### 2. Simplicity First -> 简单方案优先

原始重点：

- 只给解决当前问题所需的最小实现
- 不提前增加抽象、扩展性、配置性
- 避免把 50 行能解决的问题写成 200 行

在本 skill 里的映射：

- 只输出支撑当前决策所需的最小方案
- 不把轻量需求膨胀成重型路线图
- 不为未来未提出的事项提前设计复杂批次、额外角色或扩展体系

### 3. Surgical Changes -> 范围要精确

原始重点：

- 只改与任务直接相关的内容
- 不顺手重构邻近区域
- 每一行修改都要能追溯到请求

在本 skill 里的映射：

- 只规划用户这次明确提出的事项
- 相邻优化点可提示，但不混入主交付
- 方案中的每一段都要能追溯到当前需求

### 4. Goal-Driven Execution -> 用成功标准驱动方案

原始重点：

- 把模糊任务转成可验证目标
- 先定义如何验证，再实施
- 多步任务要说明每步的验证方式

在本 skill 里的映射：

- 先定义交付物、完成标准、错误判断标准
- 每个步骤都写验证方式、通过条件、不通过处理
- 复核机制不是“检查一下”，而是明确继续、返工或暂停

## 从源技能包提炼出的高价值句式

适合直接迁移到需求规划里的，不是原文，而是这些方法：

- 不要静默选择一种解释
- 不要把简单问题规划成复杂系统
- 不要顺手把相邻问题一起规划进去
- 不要只写动作，要写成功判定

## 适配后的默认行为

当用户说这些话时，优先这样处理：

- “优化一下”
  先改写成具体交付物和成功标准，再进入方案。
- “帮我梳理一下”
  先判断单需求/多需求，再决定输出总方案还是独立方案。
- “先排优先级”
  只做拆分、排序、依赖、批次，不提前展开所有细节。
- “继续拆”
  基于已有总方案往下拆，不重新发散出新体系。

## 什么时候最该触发这些护栏

- 用户表达模糊，存在多种理解
- 用户一次提了很多项，容易范围膨胀
- 执行者容易过度设计、过度拆分
- 任务要交给另一个 AI 或团队成员继续执行
