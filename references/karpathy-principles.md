# Karpathy Principles For Planning

这份 skill 吸收了 Karpathy 风格里最适合需求规划的三条原则，并把它们从“编码护栏”翻译成“写方案护栏”。

## 1. 写方案前先思考

原始重点：

- 不要妄下断言
- 不要掩饰困惑
- 坦诚地权衡利弊
- 明确陈述假设
- 有歧义时提出多种解释
- 必要时提出异议
- 困惑时停下来请求澄清

在本 skill 里的映射：

- 方案开头必须写判断依据和关键假设
- 信息不足时列出已知信息、缺失信息、默认假设和风险
- 如果多种解释会影响方案方向，先列选项，再给默认建议

## 2. 简单至上

原始重点：

- 用最少的代码方案解决问题
- 不增加要求之外的功能
- 不为一次性代码做抽象
- 不增加未要求的灵活性或可配置性
- 不为不可能情况增加错误处理
- 能从 200 行缩到 50 行，就重写

在本 skill 里的映射：

- 优先输出最小可交付方案
- 不把轻量需求膨胀成复杂路线图
- 不为未来可能需求提前设计复杂批次、抽象层或扩展项
- 如果 15 个步骤可以缩成 5 个步骤，就重写任务切片

## 3. 目标驱动型执行

原始重点：

- 定义成功标准
- 循环直至验证通过
- 把紧迫任务转化为可验证目标

在本 skill 里的映射：

- 先定义交付物、完成标准和错误判断标准
- 每个步骤都必须有验证方式和通过条件
- 多步骤任务尽量写成 `Step -> verify: check`
- 对 Bug、重构、验证类任务，用可复现检查或测试结果作为成功标准

## 转换示例

| 模糊说法 | 转换为 |
| --- | --- |
| 添加验证 | 编写针对无效输入的测试，然后确保它们都能通过 |
| 修复漏洞 | 编写一个能够重现该问题的测试，然后让它通过 |
| 重构 X | 确保重构前后测试均通过 |
| 优化体验 | 定义体验改善指标，并说明如何验证 |
| 梳理需求 | 判断单/多需求，定义交付物和成功标准 |

## 使用时机

最应该触发这些护栏的场景：

- 用户表达模糊
- 存在多种合理解释
- 需求容易被过度规划
- 任务要交给另一个 AI 或团队成员继续执行
- 用户要求“先分析”“先给方案”“拆成步骤”“排优先级”
