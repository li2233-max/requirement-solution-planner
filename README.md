# Requirement Solution Planner

一个面向 Codex / AI 助手的通用 skill，用来先做需求规划，再进入执行。

它的目标不是直接写代码，而是把模糊请求整理成可交付、可验证、可复核的结构化方案，适合“先出方案再实施”的工作流。

## 这个 skill 做什么

- 判断输入是单需求还是多需求
- 定义交付物、完成标准、错误判断标准、范围边界
- 把任务拆成单步可验证的执行切片
- 为每一步补上复核机制
- 多需求场景下先生成总方案，再继续拆成独立事项方案

## 适用场景

- 产品需求拆解
- 功能设计与实施规划
- 多项优化建议排序
- Bug 修复规划
- 重构计划整理
- 文档整理与路线图输出

## 强触发词

这类说法通常都应该触发这个 skill：

- “先别做，先帮我梳理一下”
- “先给我一份方案”
- “先分析再做”
- “先别写代码，先拆需求”
- “拆成可执行步骤”
- “给我一个实施方案 / 技术方案”
- “做个路线图 / 阶段计划 / 任务清单”
- “这些需求怎么排优先级 / 怎么分批 / 怎么排期”
- “拆成里程碑”
- “按总方案继续拆”

常见触发方式：

- “先别做，先帮我拆一下”
- “先给我一份方案”
- “这些需求先排个优先级”
- “根据总方案继续拆”

## 这个 skill 的特点

- 不偷做关键假设，先显式说明判断依据和默认假设
- 不把小需求膨胀成大工程，优先输出最小可交付方案
- 不顺手扩大范围，方案内容尽量只覆盖当前请求
- 不只写动作，还会写成功标准和验证方式

这些约束部分吸收了 Andrej Karpathy 风格的高价值原则，但已经翻译成更适合“需求规划”场景的形式。

## 输出长什么样

单需求时，通常会输出：

- 需求模式判断
- 交付物定义
- 完成标准与错误判断标准
- 任务切片
- 复核机制

多需求时，通常会分两阶段：

1. 先输出总方案
2. 再把 `R1 / R2 / R3 ...` 拆成独立方案

## 使用方式

把这个 skill 放进 Codex 可发现的 skills 目录后，在需要做需求规划时触发它即可。

典型流程：

1. 用户给出模糊请求
2. skill 先判断单需求 / 多需求
3. 输出结构化方案
4. 用户决定继续拆分、复核，或开始执行某一步

## 示例输入

```text
先别写代码，先帮我把这个需求拆成可执行步骤。
```

```text
我有首页改版、登录报错优化、夜间模式和文档整理，先帮我排优先级并给路线图。
```

```text
根据这份总方案继续拆，把 R1 到 R3 分别展开成独立实施方案。
```

## 设计原则

这个 skill 的第一责任不是“尽快开始做”，而是先把事情整理成：

- 可交付
- 可验证
- 可复核

这样后续无论交给另一个 AI，还是交给团队成员执行，衔接都会更顺。
