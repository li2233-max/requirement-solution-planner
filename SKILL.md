# Requirement Solution Planner

先把输入整理成可交付、可验证、可复核的方案，再决定是否进入执行。

## 执行顺序

始终按这个顺序处理：

1. 判断是单需求还是多需求
2. 定义交付物
3. 拆成可验证步骤
4. 为每一步补复核机制

如果是多需求，先补一层“拆分、分类、排序、依赖、批次规划”。

## 基本规则

- 不要在未完成结构化之前直接开始写代码、PRD、SQL、文案或实现细节，除非用户明确要求执行某一步。
- 不要接受“优化一下”“完善一下”这类模糊交付。必须改写成明确交付物、完成标准、错误判断标准、范围边界。
- 如果存在多个合理解释，不要静默选择其中一个；要显式写出判断依据、默认假设或备选路径。
- 每个步骤必须可独立验证，且一次只做一件事。
- 每完成一步都要给出复核结论：`继续`、`返工` 或 `暂停`。
- 信息不完整时，不要卡住；先列出已知信息、缺失信息、默认假设、假设风险，再给临时方案。

## 高频触发说法

以下表达通常都应触发这个 skill：

- “先别做，先帮我梳理一下”
- “先给我一份方案”
- “先分析再做”
- “先别写代码，先拆需求”
- “拆成可执行步骤”
- “给我一个实施方案 / 技术方案 / 路线图”
- “这些需求怎么排优先级 / 怎么分批 / 怎么排期”
- “帮我拆成里程碑 / 任务清单 / 阶段计划”
- “按总方案继续拆”
- “这是一件事还是多件事”

## Karpathy 风格护栏

把下面四条当作这个 skill 的行为约束：

### 1. 先暴露假设，再输出方案

- 不要替用户偷偷决定范围、优先级、实现路径或交付形式。
- 如果不同解释会导向不同方案，先列出选项、差异、影响，再给默认建议。
- 如果信息不足但仍可推进，把关键假设写进方案开头，而不是藏进后文。

### 2. 简单方案优先

- 优先给出最小可交付方案，不为未来可能出现的需求提前设计复杂批次、抽象层或扩展位。
- 不把一句话需求膨胀成厚重路线图；能用 5 个步骤讲清，就不要写成 15 个步骤。
- 如果更简单的拆法足以支撑执行，应主动选择更简单的版本。

### 3. 范围像手术刀一样精确

- 只规划用户当前请求直接需要的事项。
- 相邻机会点可以提示，但不要混入本次主方案。
- 方案里的每一段都应该能追溯到用户请求；追溯不了的内容应删掉或降级为备注。

### 4. 用成功标准驱动，而不是用动作词驱动

- 把“优化一下”“修一下”“梳理一下”改写成可验证的目标和检查方式。
- 对 Bug、性能、重构、体验类请求，明确“如何判断已经改善”。
- 多步任务尽量写成 `步骤 -> 验证` 的形式，让执行者知道什么时候算完成。

## 单需求与多需求判断

优先判断输入属于哪一类：

### 单需求

满足任一特征时，按单需求处理：

- 只有一个明确目标
- 只有一个主要交付物
- 只有一条核心执行链路
- 其余内容只是补充说明，不构成独立任务

### 多需求

满足任一特征时，按多需求处理：

- 一次提出多个功能、优化、修复、重构或文档事项
- 同时存在多个独立交付物
- 各事项可以拆成不同执行链路
- 用户明确要求排优先级、分批次、看依赖、继续拆分

如果边界模糊，默认先按多需求拆分，再决定是否合并。

输出开头必须显式写清：

- 需求模式：`单需求` / `多需求`
- 判断依据
- 是否存在独立事项

## 边界案例

遇到下面这些情况时，按以下方式处理：

- 一个目标，但包含多个明显独立交付物
  按多需求处理。
- 一个目标，只有一条主执行链路，但步骤较多
  仍按单需求处理，不要因为步骤多就误判为多需求。
- 同时包含“功能 + 文档 + 测试 + 上线”
  如果这些只是同一事项的配套动作，按单需求；如果能拆成独立负责人或独立交付物，按多需求。
- 用户丢来很多想法，没有明确主目标
  先按多需求拆分，并在开头标注“目标尚未收敛”。
- 用户说“顺手优化一下”“有空也一起做”
  默认降级为备注或候选事项，除非它已经构成独立交付物。
- 用户只让你判断需不需要拆开
  只做模式判断和拆分依据，不展开完整方案。

## 单需求模式

单需求输出必须覆盖以下部分：

- `0. 需求模式判断`
- `1. 定义交付物`
- `2. 任务切片`
- `3. 复核机制`

字段模板见 [references/templates.md](references/templates.md) 的“单需求模板”。

### 单需求约束

- 先定义目标交付物，再写步骤
- 完成标准与错误判断标准必须同时存在
- 范围边界里要明确包含项、不包含项、完成截止点
- 每个 Step 都要包含：目标、输入、动作、输出、验证方式、通过条件、不通过处理

## 多需求模式

多需求必须采用“两阶段产出”：

### 阶段 A：总方案

先生成多需求总方案，用于：

- 拆分独立事项
- 标记类型、优先级、依赖、风险
- 检查冲突与重叠
- 规划批次与总体路线图
- 给出每个事项的简要交付物定义

### 阶段 B：独立方案

在总方案基础上，把每个 `R1 / R2 / R3 ...` 当成单需求，分别展开成独立方案。

不要在阶段 B 重新发散出新的事项体系；独立方案必须继承总方案里的目标、边界、优先级、依赖关系。

字段模板见 [references/templates.md](references/templates.md) 的“多需求模板”。

## 用户意图映射

按用户表达选择输出深度：

- 用户说“给我一份方案”“先别做，先拆解”“先规划”
  直接输出完整方案；单需求给单需求方案，多需求默认先给阶段 A 总方案。
- 用户说“先做技术方案”“先做实施方案”“先给路线图”“拆成里程碑”
  默认输出完整规划，不直接进入实现。
- 用户说“这是一个需求还是多个需求”“先帮我判断需不需要拆开”
  只做模式判断、事项拆分和判断依据，不展开完整方案。
- 用户说“先排优先级”“这些先做哪个”“怎么分批”
  优先输出多需求模式中的拆分、优先级与批次规划。
- 用户说“先排期”“先列阶段计划”“先出任务清单”
  优先输出拆分、批次、依赖和推荐顺序。
- 用户说“根据总方案继续拆”“把 R1 到 R6 分别展开”
  直接进入多需求阶段 B，不重新生成总方案。
- 用户说“开始做第 1 步”“执行 R2”“继续下一步”
  只执行被指定的步骤或事项，并在结尾补复核。
- 用户说“复核一下”“继续前先检查”
  只做复核，不继续执行。

## 输出风格

- 使用简洁、结构化中文
- 多用短标签和直接表达
- 避免空话、套话、泛泛管理术语
- 让输出可以被另一个 AI 或人直接接手执行

## 质量红线

不要出现以下情况：

- 未判断单需求/多需求就直接开始规划
- 把多个事项强行写成一条单链路
- 多需求下只列 `R1/R2/R3` 概要就结束
- 未定义交付物就直接给最终实施内容
- 一个步骤塞入多个动作
- 给出不可验证的步骤
- 不做复核就进入下一步
- 已有总方案却不基于它继续拆独立方案

## 功能开发类补充要求

如果需求最终指向功能开发、界面实现或可交互流程交付，在收尾前补一条验收要求：

- 使用 `chrome-devtools` 按真实用户链路完成至少一轮自测
- 根据自测结果修正关键问题后，再给出最终结论
