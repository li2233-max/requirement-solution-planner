# 多需求方案

## 0. 需求模式判断

- 需求模式：多需求
- 判断依据：
  - 本次不是单一功能点，而是从“高频重度用户”角度对项目进行整体体验审视
  - 可以明确拆成多个独立事项，每个事项都有独立交付物和独立执行链路
  - 输入同时包含功能增强、体验优化、安全改进和产品化能力补足
- 拆分后的事项数量：6
- 是否包含优化建议：是

## 1. 需求拆分

### 1.1 独立事项清单

- R1：补齐多轮对话与会话管理能力
- R2：补齐生产可用的安全链路与内容消毒
- R3：补齐失败处理、重试与请求状态反馈
- R4：补齐用户设置与偏好持久化
- R5：优化移动端与高频使用场景下的交互效率
- R6：补齐对话资产管理能力

### 1.2 事项分类

- R1：类型 = 新功能；目标 = 让用户能连续对话而不是每次单轮重来；交付物 = 会话列表 + 消息列表 + 上下文对话能力
- R2：类型 = 架构优化 / 安全优化；目标 = 让项目从 demo 更接近可上线结构；交付物 = 服务端代理方案 + Markdown 安全渲染方案
- R3：类型 = 体验优化；目标 = 减少失败后的挫败感；交付物 = 错误状态、重试、重新生成、请求过程反馈
- R4：类型 = 体验优化；目标 = 让高频用户不用反复配置；交付物 = 模型/主题/快捷键等偏好持久化
- R5：类型 = 交互优化；目标 = 让移动端和连续提问更顺手；交付物 = 响应式输入区、操作区和可访问性优化
- R6：类型 = 新功能 / 体验优化；目标 = 让结果能沉淀、导出、复用；交付物 = 导出、分享、提示词管理、历史资产能力

### 1.3 依赖与冲突检查

- 依赖关系：
  - R1 依赖 R3 的错误处理能力，否则多轮对话失败时恢复体验很差
  - R6 依赖 R1 的会话模型，否则没有稳定的导出对象
  - R5 会受 R1 页面结构改造影响，建议一起设计
  - R2 中的内容消毒应在 R1 之前或同时完成，因为多轮后渲染风险更高
- 可并行项：
  - R2 与 R4 可并行
  - R3 与 R5 可并行
- 冲突项：
  - 暂无硬冲突
- 建议归并项：
  - R1 与 R6 在数据结构设计阶段一起考虑
  - R3 与 R5 在 UI 状态设计阶段一起考虑

## 2. 优先级与批次规划

### 2.1 优先级排序

- P0：R2 安全链路与内容消毒、R1 多轮对话与会话管理
- P1：R3 失败处理与状态反馈、R4 用户设置与偏好持久化、R5 移动端与高频使用交互优化
- P2：R6 对话资产管理能力

### 2.2 推荐实施顺序

- 第 1 批：R2
- 第 2 批：R1
- 第 3 批：R3 + R4
- 第 4 批：R5
- 第 5 批：R6

### 2.3 排序理由

- 用户价值：
  - 高频用户最痛的点是“不能连续聊”和“失败后要从头来”
  - 安全问题虽然用户表面不一定第一时间感知，但一旦出事影响最大
- 实现成本：
  - R2 和 R4 相对边界清楚
  - R1 改动面最大，要在主数据模型确定后推进
- 风险：
  - 当前前端直连密钥和 `v-html` 渲染属于高风险项
  - 多轮会话会放大性能、状态和渲染问题
- 依赖关系：
  - R6 必须建立在会话模型之上

## 3. 每项需求的交付物定义

### R1

- 交付物：多轮会话系统方案与前端实现清单
- 完成标准：
  - 用户可以创建、切换、删除会话
  - 每个会话能保存用户消息和模型回复
  - 发送消息时能带上可配置长度的上下文
  - 会话关闭页面后仍可恢复
- 错误判断标准：
  - 仍然只有单轮问答
  - 切换会话时消息错乱或丢失
  - 多轮对话导致明显卡顿或渲染错位
- 范围边界：
  - 包含前端会话模型、列表、存储、UI 交互
  - 不包含账号体系和云端同步
  - 完成截止点：本地单用户可稳定使用

### R2

- 交付物：生产安全基线改造方案
- 完成标准：
  - 模型密钥不再暴露在纯前端调用链中
  - Markdown HTML 输出经过消毒
  - 对外调用链路可切到服务端代理或 BFF
- 错误判断标准：
  - `.env` 中的密钥仍直接暴露给浏览器请求
  - 模型输出中的 HTML/脚本仍可直接注入页面
  - 生产部署仍只能依赖开发代理
- 范围边界：
  - 包含接口代理架构和前端渲染安全
  - 不包含完整后端业务系统
  - 完成截止点：具备基础上线安全前提

### R3

- 交付物：失败恢复与请求状态体系
- 完成标准：
  - 用户能明确看到“请求中 / 排队中 / 失败 / 已完成”
  - 失败后可一键重试或重新生成
  - 中断请求后状态能正确回收
- 错误判断标准：
  - 失败只出现泛化 toast，无法恢复
  - 用户不知道当前是网络错误、限流还是模型中止
  - 中断后 UI 停留在错误状态
- 范围边界：
  - 包含状态提示、重试、恢复逻辑
  - 不包含复杂监控后台
  - 完成截止点：失败可诊断、可恢复

### R4

- 交付物：用户偏好中心
- 完成标准：
  - 模型选择、主题、快捷键偏好、常用提示词能持久化
  - 刷新页面后保留上次使用习惯
  - 提供最基础设置入口
- 错误判断标准：
  - 每次刷新后设置全部丢失
  - 明明有暗色主题逻辑但用户无法使用
  - 设置入口过深或不可发现
- 范围边界：
  - 包含本地持久化和设置 UI
  - 不包含跨设备同步
  - 完成截止点：单设备本地可恢复

### R5

- 交付物：高频使用交互优化方案
- 完成标准：
  - 移动端输入区、操作按钮、滚动区域更稳定
  - 桌面端连续提问路径更短
  - 关键操作具备更好的可访问性
- 错误判断标准：
  - 小屏下输入区拥挤或按钮遮挡内容
  - 键盘交互与按钮状态不一致
  - 高强度使用下频繁需要重复点击和重复配置
- 范围边界：
  - 包含布局与操作流优化
  - 不包含整套视觉重设计
  - 完成截止点：桌面与移动端主流程顺畅

### R6

- 交付物：对话资产沉淀能力
- 完成标准：
  - 支持导出对话
  - 支持复制整轮或整会话结果
  - 支持保存常用提示词模板
- 错误判断标准：
  - 结果只能看不能管理
  - 优质回答无法复用
  - 高频用户需要手动复制粘贴到外部工具整理
- 范围边界：
  - 包含导出、复制、模板
  - 不包含团队协作与云端知识库
  - 完成截止点：个人用户可积累与复用内容

## 4. 每项需求的任务切片

### R1 的步骤

- Step 1：
  - 步骤目标：定义会话和消息的数据模型
  - 输入：当前 `src/views/chat.vue`、`src/store/business/index.ts`、`src/components/MarkdownPreview/index.vue`
  - 动作：抽象 `Conversation`、`Message`、`RequestState` 结构，明确会话列表与当前会话状态
  - 输出：数据结构文档和 store 改造草案
  - 验证方式：确认数据结构能覆盖创建、切换、追加消息、恢复会话
  - 通过条件：不再依赖单一 `inputTextString + outputTextReader`
  - 不通过处理：继续拆分消息态和会话态
- Step 2：
  - 步骤目标：把单轮 UI 改成消息流 UI
  - 输入：当前 `src/views/chat.vue`
  - 动作：从单个输出面板改成消息列表 + 输入区
  - 输出：页面结构设计稿或组件拆分方案
  - 验证方式：确认至少支持一问一答多轮展示
  - 通过条件：消息项可独立渲染且可扩展
  - 不通过处理：拆成会话列表区和消息区分步完成
- Step 3：
  - 步骤目标：加入本地持久化
  - 输入：会话 store 方案
  - 动作：使用 Pinia + localStorage 做会话缓存
  - 输出：本地恢复策略
  - 验证方式：刷新后会话仍在
  - 通过条件：当前会话和最近会话可恢复
  - 不通过处理：先只保存最近会话

### R2 的步骤

- Step 1：
  - 步骤目标：识别现有高风险点
  - 输入：`.env`、`vite.config.ts`、`src/components/MarkdownPreview/plugins/markdown.ts`
  - 动作：列出前端直连密钥、开发代理依赖、`html: true` + `v-html` 等风险
  - 输出：风险清单
  - 验证方式：逐项映射到代码位置
  - 通过条件：每项风险都有明确文件落点
  - 不通过处理：补充代码证据
- Step 2：
  - 步骤目标：设计服务端代理/BFF 接口
  - 输入：当前各模型 `chatFetch` 逻辑
  - 动作：统一成后端转发协议
  - 输出：接口契约草案
  - 验证方式：前端不再直接依赖第三方密钥
  - 通过条件：至少可覆盖主推模型
  - 不通过处理：先只代理 DeepSeek 与 Moonshot
- Step 3：
  - 步骤目标：加上 HTML 消毒
  - 输入：`dompurify` 依赖和 Markdown 渲染链路
  - 动作：在渲染前引入 DOMPurify 消毒输出 HTML
  - 输出：安全渲染方案
  - 验证方式：恶意 HTML 不能直接执行
  - 通过条件：保留 Markdown 能力同时阻断注入
  - 不通过处理：先默认关闭原始 HTML 渲染

### R3 的步骤

- Step 1：
  - 步骤目标：定义请求生命周期状态
  - 输入：当前 `readerLoading`、`waitingForQueue`、`isAbort`
  - 动作：统一成状态机
  - 输出：状态枚举和切换表
  - 验证方式：每次请求只有一个明确状态
  - 通过条件：状态无歧义
  - 不通过处理：减少状态数量
- Step 2：
  - 步骤目标：补 retry / regenerate
  - 输入：发送逻辑与最后一次 prompt
  - 动作：保留 last prompt，提供重试和重新生成
  - 输出：恢复入口
  - 验证方式：失败后可一键继续
  - 通过条件：不需要重新手工粘贴问题
  - 不通过处理：先只实现 retry

### R4 的步骤

- Step 1：
  - 步骤目标：梳理可持久化偏好项
  - 输入：当前模型选择、`useTheme.ts`、快捷键逻辑
  - 动作：确认哪些项进入设置中心
  - 输出：设置项清单
  - 验证方式：覆盖高频用户常改项
  - 通过条件：设置项少而有用
  - 不通过处理：先只保留模型、主题、发送快捷键
- Step 2：
  - 步骤目标：实现设置存储与恢复
  - 输入：Pinia store 设计
  - 动作：添加偏好 store 和 localStorage 持久化
  - 输出：可恢复设置
  - 验证方式：刷新页面后保持上次设置
  - 通过条件：主要偏好项恢复成功
  - 不通过处理：分项落地

### R5 的步骤

- Step 1：
  - 步骤目标：梳理当前移动端与高频操作痛点
  - 输入：`src/views/chat.vue`、`src/components/MarkdownPreview/index.vue`
  - 动作：定位固定宽度、浮动按钮、滚动区域和输入区拥挤问题
  - 输出：交互问题清单
  - 验证方式：桌面和移动端路径都可复述
  - 通过条件：问题有代码落点
  - 不通过处理：先只处理输入区与发送按钮
- Step 2：
  - 步骤目标：优化操作流
  - 输入：问题清单
  - 动作：把“切模型、输入、发送、停止、复制、重新生成”组织成更顺手的布局
  - 输出：新版交互草图
  - 验证方式：主流程点击次数减少
  - 通过条件：连续提问更顺手
  - 不通过处理：先优化移动端

### R6 的步骤

- Step 1：
  - 步骤目标：定义导出与模板能力边界
  - 输入：当前复制能力、Markdown 输出
  - 动作：明确支持导出 markdown / txt、保存 prompt 模板
  - 输出：资产能力最小集合
  - 验证方式：高频用户常见复用场景被覆盖
  - 通过条件：至少覆盖导出和模板
  - 不通过处理：先只做导出
- Step 2：
  - 步骤目标：挂接到会话系统
  - 输入：R1 会话数据结构
  - 动作：按会话或消息粒度提供导出和复制
  - 输出：资产操作入口
  - 验证方式：用户能管理而不是只能阅读结果
  - 通过条件：操作清晰
  - 不通过处理：先支持当前会话导出

## 5. 总体复核机制

### 单项复核

- R1 是否让产品从“单轮 demo”升级为“可连续使用的聊天工具”
- R2 是否显著降低了前端密钥暴露和内容注入风险
- R3 是否让失败后的体验从“中断”变成“可恢复”
- R4 是否减少了高频用户重复配置成本
- R5 是否改善了移动端和连续提问时的顺滑度
- R6 是否让优质输出可沉淀、可复用

### 整体验收

- P0 是否全部完成：
  - R2 与 R1 是本轮最关键验收项
- 是否存在未解决的依赖阻塞：
  - 若无统一会话模型，不应进入 R6
- 多项需求合并后是否仍满足最初目标：
  - 应让项目从“前端聊天 MVP”向“可高频使用的轻量 AI 工作台”演进
- 最终建议：
  - 先做子集：R2 -> R1 -> R3/R4
- 功能开发类补充验收：
  - 功能完成后，必须再使用 `chrome-devtools` 按真实交互链路自测一遍

## 6. 源码分析依据

以下判断直接来自当前代码结构：

- 当前产品仍以单轮对话为核心：
  - `README.md` 明确写了“仅支持单轮对话模式”
  - `src/views/chat.vue` 只有单个 `inputTextString` 和单个输出 reader，没有消息数组和会话列表
- 模型调用仍是纯前端链路：
  - `src/components/MarkdownPreview/models/index.ts` 直接从 `import.meta.env.*` 取密钥并发起请求
  - `vite.config.ts` 的代理主要服务开发环境，不是正式后端代理方案
- Markdown 安全链路不完整：
  - `src/components/MarkdownPreview/plugins/markdown.ts` 开启了 `html: true`
  - `src/components/MarkdownPreview/index.vue` 使用 `v-html`
  - 项目依赖里有 `dompurify`，但源码里没有实际使用
- 高级用户偏好没有持久化：
  - 没有看到围绕模型、主题、快捷键的 `localStorage` 或持久化 store
  - `src/hooks/useTheme.ts` 暗色主题变量已准备，但页面没有可见切换入口
- 错误恢复能力较弱：
  - `src/store/business/index.ts` 失败只回 `error: 1`
  - `src/views/chat.vue` 的失败处理仍偏向 toast 提示
- 结果沉淀能力有限：
  - 当前更偏“展示结果”，只有复制，没有会话级导出和模板管理

## 7. 最终结论

从忠实高频用户的角度看，这个项目现在最强的是“轻、快、上手低、模型流式渲染链路清楚”；最明显的短板是“还停留在单轮 demo 体验”，距离真正高频使用的 AI 工具还差三块能力：

1. 可持续对话
2. 可恢复与可配置
3. 可上线与可沉淀

如果只做最关键的一轮升级，建议优先推进：

1. R2 安全链路与内容消毒
2. R1 多轮对话与会话管理
3. R3 失败恢复与重新生成
